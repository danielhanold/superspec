# Promote `apply` to a Schema Artifact and Fix `verify.requires`

**Date:** 2026-05-16
**Schema:** `superspec`
**Target version bump:** `1 → 2`

## Problem

The current schema declares `verify.requires: [plan]`, with an inline comment stating that this edge "exists only for schema graph purposes" and the *real* prerequisite is the completed apply phase. Because the OpenSpec CLI's artifact DAG is built strictly from the `artifacts:` list, and the `apply:` block is a separate top-level configuration with no `generates` field, the schema has no way to express "verify depends on apply having run."

Agents reading the schema follow the DAG and routinely run `/opsx:verify` before `/opsx:apply` has executed, because the DAG says verify is reachable as soon as plan is complete. The inline comment is invisible to the resolver and easy for an agent to skip past.

## Goals

- Make the schema graph honestly express `plan → apply → verify`.
- Cause `/opsx:verify` to be unreachable until an apply artifact exists.
- Keep `/opsx:apply` CLI behavior unchanged.
- Document the `apply → verify → repeat` convergence pattern that this restructuring enables (the new command that automates the loop is intentionally deferred to a follow-up change).

## Non-goals

- Building any `/opsx:apply-loop` or auto-iteration command.
- Modifying `/opsx:ff` to also run apply.
- Adding lint/tooling to detect drift between the `apply:` block instruction and the apply artifact's redirect.

## Approach

Hybrid: the apply phase is represented twice in the schema — once as a real artifact (DAG node) and once as the existing top-level `apply:` block.

**Why hybrid (rejected alternatives):**

- **Pure-artifact (drop the `apply:` block)** — rejected. The `/opsx:apply` command handler defaults `applyConfig?.requires` to `schema.artifacts.map(a => a.id)`, i.e. *all* artifacts. Removing the `apply:` block would silently make `/opsx:apply` require `verify` to exist before it could run, creating an impossible chicken-and-egg.
- **Upstream OpenSpec change (add `generates` to `ApplyPhaseSchema`)** — rejected. Depends on upstream timeline; this fix is feasible inside Superspec's own schema today.

## CLI behavior dependencies (the constraints driving the design)

Verified against Fission-AI/OpenSpec `main` (`src/core/artifact-graph/schema.ts`, `types.ts`; `src/commands/workflow/instructions.ts`):

1. `validateRequiresReferences` rejects any `requires:` entry that does not match an `artifacts[*].id`. Therefore `verify.requires: [apply]` only validates if `apply` is in `artifacts:`.
2. `ApplyPhaseSchema` has only `requires`, `tracks`, `instruction`. No `generates` field; the apply phase cannot participate in completion detection on its own.
3. `detectCompleted` walks `schema.artifacts` and checks `generates` file existence. Only artifact nodes can mark themselves complete.
4. `generateApplyInstructions` (the `/opsx:apply` handler) reads `schema.apply.instruction` directly. It does *not* read the apply artifact's instruction body. The `apply:` block is the canonical text surface for `/opsx:apply`.
5. `/opsx:continue` walks the artifact DAG via `getNextArtifacts(completed)` and surfaces each artifact's own `instruction`.

The design respects all five constraints.

## Schema changes (machine-readable diff sketch)

```yaml
name: SuperSpec
version: 2                 # was 1
description: |
  ... existing description ...
  v2: apply is now a DAG node generating apply.md. verify.requires now
  points at apply rather than plan, so /opsx:verify cannot run before
  /opsx:apply has executed. Convergence loop (apply → verify → repeat
  on fixable failures) is documented in workflow-details.md.

artifacts:
  # ... brainstorm, proposal, design, specs, tasks, plan unchanged ...

  - id: apply
    generates: apply.md
    description: Implementation receipt confirming the apply phase completed
    template: apply.md
    instruction: |
      This artifact is generated by `/opsx:apply`, not `/opsx:continue`.
      If you reached this via `/opsx:continue`, exit and run `/opsx:apply`
      instead. The canonical implementation instructions live in the
      `apply:` phase block below.

      apply.md is a minimal receipt written at the end of the apply
      phase: change name, iteration counter, applied-at timestamp,
      executor, worktree path, branch, commit range, tasks completed
      X of Y, and remaining tasks. Verification outcome is recorded in
      verify.md (Overall Decision section), not here.

      If apply.md already exists when re-entering apply, read the
      previous iteration value and increment by one; otherwise default
      to iteration 1.
    requires:
      - plan

  - id: verify
    generates: verify.md
    description: Post-implementation verification against specs, design, tasks, code
    template: verify.md
    instruction: |
      Use the Skill tool to invoke openspec-verify-change (the
      `/opsx:verify` slash command is its user-facing equivalent).

      apply.md MUST exist before this artifact runs — the DAG enforces
      this. If apply.md is missing, run `/opsx:apply` first. After every
      apply iteration (whether the first or a re-run), verify should be
      invoked so the convergence loop can decide whether more apply
      iterations are needed.

      verify.md is overwritten each run. Include the iteration counter
      from apply.md so verify and apply stay in lockstep.

      The five checks are unchanged from v1:
      1. Structural validation (openspec validate --all --json)
      2. Task completion (all - [ ] → - [x] in tasks.md)
      3. Delta spec sync state
      4. Design/specs coherence (non-blocking warning)
      5. Implementation signal (no unstaged files)

      Recommended convergence loop:
      - PASS → proceed to finishing-a-development-branch + archive.
      - PASS_WITH_WARNINGS → proceed; record warnings.
      - FAIL with code-fixable items → return to apply, re-run, re-verify.
      - FAIL with artifact-level items (spec drift) → fix the offending
        artifact, re-run from apply.
      - Iteration > 5 → stop looping; report to the user.

      See docs/workflow-details.md for the full convergence pattern.
    requires:
      - apply         # was: plan

apply:
  requires: [plan]    # unchanged; /opsx:apply still gates on plan
  tracks: tasks.md
  instruction: |
    # The canonical apply-phase body. (Current v1 instruction text
    # preserved here: pre-flight, worktree, executor 2a/2b, verification,
    # finishing-branch.) Plus, at the end of the executor step, a new
    # short paragraph instructing the agent to write apply.md per the
    # template, incrementing the iteration counter if a previous
    # apply.md exists.
```

The body of the existing `apply:` instruction is preserved verbatim, with a single new step appended:

> **Receipt**: After step 3 (Verification) completes, write `apply.md`
> following `templates/apply.md`. If a previous `apply.md` exists in
> the change directory, read its `Iteration:` field and increment by
> one; otherwise set iteration to `1`. Overwrite the file.

The `apply:` block is therefore the single source of truth for the implementation body; the apply artifact's instruction is a short redirect that exists only so `/opsx:continue` walkers do not silently fail.

## New file: `templates/apply.md`

```markdown
# Apply Receipt

> Generated at the end of the apply phase to mark code-implementation
> complete and provide verify with the state it needs.
> Overwritten on each apply iteration; iteration counter grows.

**Change**: `<change-name>`
**Iteration**: `1`
**Applied at**: `YYYY-MM-DD HH:mm`
**Executor**: `subagent-driven-development` | `executing-plans`

---

## Workspace

- **Worktree**: `.worktrees/<change-name>/`
- **Branch**: `<branch-name>`

---

## Commits

- **Range**: `<from-sha>..<to-sha>` (or `none` if nothing committed yet)
- **Count**: `<n>`

---

## Tasks

- **Completed**: `X of Y` checkboxes in tasks.md flipped to `- [x]`
- **Remaining**: `<list ids of unfinished tasks, or "none">`

---

## Next step

`<e.g., "Run /opsx:verify" or "Re-run apply to address <issue>">`
```

## Modified file: `templates/verify.md`

Add an `Iteration` line directly under the existing `**Verified at**` line:

```markdown
**Iteration**: `<integer; copy from apply.md>`
```

Add a closing instruction near the "Overall Decision" section reminding the agent that FAIL with code-fixable items routes back to apply rather than to a fix in place.

## Convergence loop (the recommended pattern that documentation captures)

```text
        plan
         │
         ▼
       apply ─────► apply.md (iteration N)
         │
         ▼
       verify ────► verify.md (iteration N)
         │
         ├── PASS or PASS_WITH_WARNINGS ──► finishing-a-development-branch → archive
         │
         ├── FAIL, items fixable by code change ──► return to apply (N+1)
         │
         ├── FAIL, items in artifacts (spec drift) ──► fix artifact → apply (N+1)
         │
         └── Iteration > 5 ──► stop; report to user
```

The schema enforces the file-existence dependency. The agent (or a future loop-runner command) makes the iteration decision. No tool currently automates the loop in this change — that is the deferred follow-up.

## Testing

- `openspec validate` against the new schema parses cleanly (zod) and validates the DAG (no cycle, all `requires` resolve to artifact IDs).
- Hand-walk: scaffold `openspec/changes/_test-apply-artifact/` with empty stubs (brainstorm.md, proposal.md, specs/<cap>/spec.md, tasks.md, plan.md). Confirm:
  - `openspec status` lists `apply` as a `ready` node once plan exists.
  - `/opsx:verify` reports the change as blocked (missing artifact `apply`) when `apply.md` is absent.
  - `/opsx:apply` surfaces the `apply:` block instruction text (not the artifact redirect).
- Doc-DAG consistency: grep for `verify.requires` and `requires: [plan]` references in `docs/` and `openspec/schemas/`; every match must be updated to the new dependency or rewritten in prose.
- Template-instruction alignment: each instruction text references field names that appear in its template.

## Edge cases

- **apply.md missing iteration field** — instruction tells the agent to default to iteration 1.
- **apply.md exists, plan changed** — re-apply increments iteration; agent surfaces a warning if the receipt's commit range no longer matches the plan structure. Recommended check; no schema enforcement.
- **Re-running `/opsx:apply` after success** — produces apply.md with iteration > 1; verify still satisfied because requires are file-existence based, not content-based.
- **Iteration > 5** — verify instruction tells the agent to stop the loop pattern and report the situation to the user.

## Migration from schema v1

- `schema.version: 1 → 2`.
- INTEGRATION.md gains a short "Migration from v1" section: in-flight changes that already have a `verify.md` but no `apply.md` must author a minimal `apply.md` (use `templates/apply.md`) before re-running `/opsx:verify` under v2. The minimal author-by-hand version is: copy the template, set iteration to 1, fill the worktree path, branch, commit range, and task counts from the existing implementation.
- The Superspec repository itself has no `openspec/changes/` directory, so there is no internal migration work in this change.

## Risks

- **Drift between the `apply:` block and the apply artifact's redirect.** If a future editor adds detail to the artifact instruction without updating the redirect (or vice versa), the two will fall out of sync. Mitigation: the artifact instruction is intentionally short and contains nothing operational; reviewers should reject any PR that grows the artifact instruction beyond the redirect message.
- **External consumers using schema v1.** Pinning to v1 in their `openspec/schemas/superspec/` copy avoids the change entirely; pulling the v2 schema requires the migration note above. The schema lives in their repo (project-local schema resolution per `getSchemaDir`), so update timing is fully under their control.

## Open questions

None known after the brainstorming pass. The deferred follow-up (loop-runner command) will need its own design.

## Out of scope (for the follow-up change)

The auto-iteration command will need:
- A name (e.g. `/opsx:apply-loop` or `/opsx:converge`).
- A wrapping shell that invokes `/opsx:apply` → `/opsx:verify` → branch on the verify outcome.
- Stop-on-non-convergence logic at iteration > 5.
- Optional: aggregation of per-iteration outcomes into a `loop.md` or a single rolled-up `apply.md` history section.

That change should be planned against the schema produced by *this* change — once `verify.requires: [apply]` is in place, the iteration semantics are stable.
